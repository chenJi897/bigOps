# BigOps 工单多级审批与通知方案

**版本**: v1.0  
**日期**: 2026-03-24  
**状态**: 设计中

## 一、目标

BigOps 当前工单模块适合故障处理和事件流转，但不适合资源申请、权限申请、变更申请等需要多级审批的场景。本方案的目标是：

1. 在保留现有事件/故障工单能力的前提下，引入多级审批能力。
2. 将审批、执行、通知解耦，避免把复杂流程堆在单一 Ticket 状态机中。
3. 支撑后续与 CMDB、服务树、资源池、任务执行中心的联动。

## 二、核心原则

### 2.1 工单分层

- `incident`: 故障/事件单，不走审批，继续使用当前处理流。
- `request`: 服务请求单，走多级审批。
- `change`: 变更单，走多级审批，必要时支持 CAB。

### 2.2 审批与执行解耦

审批通过不等于执行完成。系统需要在审批完成后生成执行单，由人工或自动化能力完成资源交付、配置变更、权限开通等动作。

### 2.3 通知与业务真相解耦

审批结果、执行结果以数据库记录为准。邮件、Webhook、IM 只是通知渠道，不应成为业务状态来源。

## 三、业务模型

### 3.1 请求模板 request_template

请求模板相当于“服务目录项”。例如：

- 申请测试服务器
- 申请数据库账号
- 申请白名单
- 申请堡垒机权限
- 申请发布变更

模板负责定义：

- 展示名称
- 请求分类
- 表单 schema
- 默认审批策略
- 默认执行模板
- 是否启用

### 3.2 审批策略 approval_policy

审批策略是流程模板头。它定义某类请求如何审批，但不绑定具体实例。

审批策略下有多个审批阶段。

### 3.3 审批阶段 approval_policy_stage

每个阶段定义：

- 第几级
- 串行 / 并行
- 审批人解析方式
- 审批通过规则
- 超时时间
- 额外规则配置

审批人解析方式建议首批支持：

- `direct_manager`
- `dept_leader`
- `fixed_user`
- `fixed_role`
- `service_owner`
- `resource_pool_owner`
- `security_group`

### 3.4 审批实例 approval_instance

审批实例对应某一张请求单当前跑到的审批流程实例。核心字段：

- 当前阶段
- 整体状态
- 发起时间
- 完成时间

### 3.5 审批记录 approval_record

审批记录用于追踪每一级、每位审批人的动作。支持：

- 同意
- 拒绝
- 退回
- 转审
- 加签
- 超时

### 3.6 执行单 execution_order

审批通过后生成执行单。执行单负责：

- 记录执行类型
- 执行器来源
- 执行 payload
- 执行结果
- 执行状态

### 3.7 通知中心 notification

通知中心拆成两层：

- `notification_event`: 事件头
- `notification_delivery`: 渠道投递记录

建议额外保留站内通知模型，用于前端消息中心。

## 四、与现有 Ticket 的关系

### 4.1 Ticket 主表继续保留

现有 `tickets` 继续作为请求/事件的统一主单据表，但需要补充语义字段：

- `ticket_kind`: incident/request/change
- `request_template_id`
- `approval_status`
- `approval_instance_id`
- `execution_status`

### 4.2 现有状态机的定位

现有状态机：

- `open`
- `processing`
- `resolved`
- `rejected`
- `closed`

建议未来只用于：

- incident 处理流
- request/change 在审批通过后的执行流

审批本身不要继续复用这套状态，而应使用独立审批状态。

## 五、V1 最小实现

### 5.1 本轮范围

V1 先做：

1. 请求模板模型
2. 审批策略与审批阶段模型
3. 审批实例与审批记录模型
4. 执行单模型
5. 通知事件与投递记录模型
6. 站内通知模型
7. AutoMigrate 接入
8. 基础 Repository

### 5.2 V1 不做

- BPMN/流程引擎
- 并行复杂分支
- 加签/会签完整能力
- CAB/ECAB
- IM Bot 主动回写审批动作
- 自动化执行引擎联动

## 六、通知事件建议

首批事件：

- `request_submitted`
- `approval_pending`
- `approval_approved`
- `approval_rejected`
- `approval_timeout`
- `execution_started`
- `execution_succeeded`
- `execution_failed`
- `resource_expiring`
- `resource_reclaimed`

渠道建议：

- `in_app`
- `email`
- `webhook`
- `im`

## 七、IM 选型建议

### 7.1 Zulip

适合强调线程化协作、纯开源、自托管控制的场景。

### 7.2 Mattermost

适合运维 ChatOps、机器人和命令集成、运维协同值班。

### 7.3 Matrix + Element

适合强调开放协议、自托管和长期生态自由度的团队。

## 八、推荐实施顺序

### V1

- 后端基础模型 + Repository
- 站内通知 + 邮件 + Webhook 适配器
- 请求模板和审批策略管理 API
- 审批待办 API

### V2

- 请求提单页
- 审批待办页
- 审批流详情页
- 审批通过后自动生成执行单

### V3

- IM 适配器
- IM 卡片提醒
- 深链审批
- 催办与超时

## 十、结论

BigOps 下一步不应该继续给现有 Ticket 状态机打补丁，而应该建设“请求模板 + 审批策略 + 审批实例 + 执行单 + 通知中心”的基础设施层。这样既能满足资源申请类多级审批，也能兼容未来变更单、权限申请、自动执行和 IM 通知。
