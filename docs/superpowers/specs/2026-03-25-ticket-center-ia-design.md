# 工单管理信息架构与页面重构设计

## 1. 背景

当前工单相关能力已经具备：

- 工单列表
- 发起/创建工单
- 工单详情
- 工单类型
- 请求模板
- 审批策略
- 审批待办

但当前导航和页面语义偏后台配置视角，主要问题是：

1. 用户同时看到“工单类型”“请求模板”“审批策略”“审批待办”，认知负担大。
2. `TicketList.vue` 同时承载“全部工单 / 我创建的 / 我处理的 / 本部门”多种视角，用户入口不直观。
3. `TicketCreate.vue` 首屏同时暴露“工单类型 + 请求模板”双入口，用户需要理解内部实现概念。
4. `TicketTypes.vue` 的字段更像底层规则配置，不像用户视角下的“工单模板”。
5. `ApprovalInbox.vue` 作为二级菜单不合理，因为审批动作本质依附于具体工单，而不是独立单据中心。

本次重构的目标不是推翻现有审批与工单模型，而是把工单管理从“系统配置视角”改造成“用户使用视角”。

## 2. 设计目标

### 2.1 目标

1. 保留“工单管理”父目录。
2. 将工单管理二级菜单统一收敛为：
   - 发起工单
   - 我的待办
   - 我的申请
   - 工单模板
3. 发起工单只暴露“模板”概念，不再暴露“工单类型 + 请求模板”双入口。
4. 我的待办固定表示“我当前处理的工单”。
5. 我的申请固定表示“我创建的工单”。
6. 工单模板页面参考用户提供截图，外观和信息密度向模板管理表格靠拢。
7. 审批待办不再作为左侧菜单，审批入口收敛到工单详情和站内通知。

### 2.2 非目标

1. 不重构审批实例/审批记录后端模型。
2. 不推翻 `ticket_types` / `request_templates` / `approval_policies` 三张表。
3. 不在本轮引入新的流程引擎或状态机。
4. 不改动工单详情的核心处理动作流。

## 3. 信息架构

### 3.1 新导航结构

父目录：`工单管理`

二级菜单调整为：

1. `发起工单`
2. `我的待办`
3. `我的申请`
4. `工单模板`

从左侧主导航移除：

1. `审批待办`
2. `工单类型`
3. `请求模板`
4. `审批策略`

说明：

- `审批策略` 不删除能力，只从主导航降级为模板管理过程中的配置入口。
- `工单类型` 继续作为底层规则配置存在，但不再作为工单管理主导航入口。
- `请求模板` 将升级为“工单模板”的主页面载体。

### 3.2 页面与现有文件映射

| 新菜单 | 页面职责 | 复用文件 |
|------|------|------|
| 发起工单 | 模板入口 + 表单发起 | `TicketCreate.vue` |
| 我的待办 | 我处理的工单列表 | `TicketList.vue` |
| 我的申请 | 我创建的工单列表 | `TicketList.vue` |
| 工单模板 | 模板列表 + 模板编辑 | `RequestTemplates.vue` |
| 工单详情 | 隐藏详情页 | `TicketDetail.vue` |

## 4. 页面级设计

### 4.1 发起工单

页面标题统一改为：`发起工单`

#### 当前问题

- 第一步同时展示“工单类型卡片”和“请求模板卡片”。
- 用户需要理解“类型”和“模板”的区别。

#### 新设计

1. 第一步只展示“工单模板卡片”。
2. 用户只需要理解“选择一个模板发起工单”。
3. 模板卡片展示：
   - 模板名
   - 简短描述
   - 分类标签
4. 选择模板后进入表单页。
5. 表单页继续使用当前动态 `form_schema` 渲染逻辑。

#### 底层映射

虽然用户只看到“工单模板”，底层仍保留：

- `request_template_id`
- `ticket_kind`
- `type_id`

其中：

- `request_template` 负责用户入口
- `ticket_type` 继续作为底层默认优先级、处理部门、自动分派规则来源

### 4.2 我的待办

页面标题改为：`我的待办`

#### 当前问题

- `TicketList.vue` 同时用 tabs 承载多种视角。
- “我处理的”只是 tabs 之一，不是独立入口。

#### 新设计

1. 复用 `TicketList.vue`。
2. 固定查询视角：`scope = my_assigned`。
3. 隐藏当前 scope tabs。
4. 保留筛选条件：
   - 状态
   - 优先级
   - 工单模板/类型
   - 关键字
5. 右上角保留“发起工单”按钮。

### 4.3 我的申请

页面标题改为：`我的申请`

#### 新设计

1. 复用 `TicketList.vue`。
2. 固定查询视角：`scope = my_created`。
3. 隐藏当前 scope tabs。
4. 保留筛选条件，与“我的待办”一致。
5. 右上角保留“发起工单”按钮。

### 4.4 工单模板

页面标题改为：`工单模板`

#### 设计定位

用户给出的参考图更接近一个“模板管理页”，而不是当前 `TicketTypes.vue` 这种底层规则配置页。

因此本次选择：

- 页面外观与交互风格参考当前 `TicketTypes.vue`
- 数据主体使用当前 `RequestTemplates.vue`

即：

**工单模板 = RequestTemplate 为主，TicketType 退到底层支撑。**

#### 列表字段

保留：

1. `ID`
2. `模板名`
3. `所属类别`
4. `节点列表`
5. `启用`
6. `备注`
7. `更新时间`
8. `操作`

移除：

1. `编码`
2. `排序`
3. `执行模板`
4. `单据类型`
5. `绑定类型`

#### 字段映射

| 页面字段 | 数据来源 |
|------|------|
| 模板名 | `request_templates.name` |
| 所属类别 | `request_templates.category` |
| 节点列表 | 绑定的 `approval_policy.stages` 拼接摘要 |
| 启用 | `request_templates.status` |
| 备注 | `request_templates.description` |
| 更新时间 | `request_templates.updated_at` |

#### 节点列表生成规则

读取 `approval_policy.stages`，按顺序拼接为：

`发起 => 一级审批 => 二级审批 => 三级审批`

如果模板未绑定审批策略，则展示：

`发起 => 处理`

#### 顶部操作

保留：

1. `刷新`
2. `新增工单模板`

不作为左侧菜单保留，但仍提供入口：

1. `审批策略`

入口形式：

- 工单模板页右上角按钮
- 或模板编辑弹窗中的“管理审批策略”跳转入口

### 4.5 工单详情

`TicketDetail.vue` 继续保留，不放入左侧主导航。

保留现有结构：

1. 基本信息
2. 请求表单
3. 关联资源
4. 审批信息
5. 活动记录
6. 处理动作

审批待办菜单移除后，审批动作仍有两个自然入口：

1. 工单详情中的审批信息区
2. 站内通知跳转

## 5. 相关概念的用户视角收口

### 5.1 用户可见概念

用户视角只暴露：

- 工单模板
- 发起工单
- 我的待办
- 我的申请

### 5.2 系统内部保留概念

系统内部继续保留：

- `ticket_type`
- `request_template`
- `approval_policy`

但这三个概念不再全部直接暴露到普通操作流程中。

### 5.3 概念关系

底层关系仍然是：

`工单模板(RequestTemplate)` -> 绑定 `工单类型(TicketType)` -> 可绑定 `审批策略(ApprovalPolicy)`

用户只需要看到：

`选择工单模板 -> 填表 -> 发起工单`

## 6. 路由与菜单落地设计

### 6.1 菜单迁移目标

保留父目录：

- `工单管理`

目标子菜单：

1. `ticket_launch` -> `/ticket/create` -> `TicketCreate`
2. `ticket_todo` -> `/ticket/todo` -> `TicketList`
3. `ticket_applied` -> `/ticket/applied` -> `TicketList`
4. `ticket_templates` -> `/ticket/templates` -> `RequestTemplates`

需要移除主导航的菜单：

1. `approval_inbox`
2. `ticket_types`
3. `request_templates`
4. `approval_policies`

### 6.2 复用式路由模式

`TicketList.vue` 复用为两个路由：

- `/ticket/todo`
- `/ticket/applied`

通过 route meta 或 route path 注入页面模式：

- `mode = todo`
- `mode = applied`

页面内部据此决定：

1. 标题
2. 固定 scope
3. 是否显示 scope tabs

### 6.3 隐藏 companion routes

保留并继续由路由层补充：

1. `TicketDetail`

`TicketCreate` 将从隐藏 companion route 升级为显式菜单页，但仍可保留现有 companion 兼容逻辑，避免旧入口失效。

## 7. 旧页面处理策略

### 7.1 TicketTypes.vue

处理策略：

- 从工单管理主导航移除
- 保留后台底层配置能力
- 后续如需，可迁移到系统管理或模板内部高级配置入口

### 7.2 ApprovalInbox.vue

处理策略：

- 从工单管理主导航移除
- 暂不删除页面文件
- 后续仍可通过通知跳转或隐藏路由使用

### 7.3 ApprovalPolicies.vue

处理策略：

- 从工单管理主导航移除
- 作为工单模板页面的配置辅助入口继续保留

## 8. 迁移规则

### 8.1 数据层

本次不删除表：

- `ticket_types`
- `request_templates`
- `approval_policies`

本次主要调整：

1. 菜单数据
2. 路由入口
3. 页面标题和交互

### 8.2 菜单层

需要通过 migration/init SQL 做幂等迁移：

1. 修正旧菜单标题和路径
2. 新增新 IA 所需菜单
3. 隐藏旧入口
4. 保证旧环境升级后结构一致

1. 菜单 migration 与前端路由重构
2. `TicketList.vue` 双模式复用

4. `RequestTemplates.vue` 重构为工单模板页
5. 收口旧入口和按钮跳转

## 11. 结论

本次重构的核心不是简单改名，而是把工单管理从“后台配置视角”重构成“用户使用视角”。

最终用户只需要理解：

1. 发起工单
2. 我的待办
3. 我的申请
4. 工单模板

而底层的：

- 工单类型
- 请求模板
- 审批策略

继续作为系统支撑能力保留，但不再全部直接暴露给主使用流。
