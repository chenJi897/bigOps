# BigOps 大运维平台 - 模块设计方案

**版本**: v1.0
**日期**: 2026-03-17
**状态**: 已定稿

## 一、模块总览

BigOps平台共规划8个功能模块，按照开发优先级分为三个阶段：

### 第一阶段：基础运维能力（核心模块）
- **模块01**: 前后端底座
- **模块02**: 服务树和CMDB
- **模块03**: 自助工单

### 第二阶段：自动化能力（独立模块）
- **模块04**: 任务执行中心
- **模块05**: Prometheus监控平台
- **模块08**: 数据库和SQL管理平台

### 第三阶段：高级能力（预留）
- **模块06**: K8s多集群和APP管理（暂不实现）
- **模块07**: CICD平台和灰度发布（暂不实现）

## 二、模块01：前后端底座

### 2.1 模块概述

前后端底座是整个平台的基础设施，提供用户认证、权限管理、菜单管理、操作日志等核心功能。

### 2.2 功能清单

#### 用户管理
- 用户增删改查
- 用户状态管理（启用/禁用）
- 密码修改和重置
- 用户信息维护

#### 角色管理
- 角色增删改查
- 角色权限分配
- 用户角色关联

#### 权限管理
- 基于Casbin的RBAC模型
- API权限控制
- 数据权限控制
- 菜单权限控制

#### 菜单管理
- 菜单树形结构
- 菜单图标和路由
- 菜单权限关联
- 动态菜单加载

#### 操作日志
- 自动记录操作日志
- 日志查询和导出
- 操作审计追溯

### 2.3 技术实现

#### 后端实现

**用户认证流程**:
```
1. 用户登录 → 验证用户名密码（Bcrypt）
2. 生成JWT Token（access_token + refresh_token）
3. Token存储到Redis（支持撤销）
4. 返回Token和用户信息
```

**权限验证流程**:
```
1. 请求携带Token → JWT中间件验证
2. 解析用户信息 → 从Redis获取权限
3. Casbin验证权限 → 允许/拒绝
4. 记录操作日志（异步）
```

**数据模型**:
- users: 用户表
- roles: 角色表
- permissions: 权限表
- user_roles: 用户角色关联表
- role_permissions: 角色权限关联表
- menus: 菜单表
- operation_logs: 操作日志表

#### 前端实现

**登录页面**:
- 用户名密码登录
- 记住密码（本地存储）
- 登录状态保持

**布局结构**:
- 侧边栏导航（可折叠）
- 顶部导航栏（用户信息、退出）
- 标签页（多页面切换）
- 面包屑导航

**权限控制**:
- 路由守卫（页面级权限）
- v-permission指令（按钮级权限）
- 动态菜单（根据权限加载）

### 2.4 API接口

- 记录资产变更历史

#### 后端实现

```go
    UpdatedAt time.Time

```

**CMDB数据模型（固定字段方案）**:
```go
type Asset struct {
    ID           uint   `gorm:"primarykey"`
```

- 节点搜索和过滤

**CMDB表格**:

**资产详情**:

#### 任务管理

- 目标机器选择

- 超时控制

#### 实时监控
- 实时日志输出
- 执行进度展示
- 执行结果统计

### 5.3 技术实现

#### 架构设计

```
┌─────────────┐         ┌─────────────┐
│   Web UI    │         │   Server    │
│  (前端)      │◄───────►│  (Gin+gRPC) │
└─────────────┘         └──────┬──────┘
                               │ gRPC
                    ┌──────────┼──────────┐
                    ▼          ▼          ▼
              ┌─────────┐ ┌─────────┐ ┌─────────┐
              │ Agent 1 │ │ Agent 2 │ │ Agent N │
              └─────────┘ └─────────┘ └─────────┘
```

#### Server端实现

**任务调度**:
- HTTP API接收任务请求
- 任务入队（Asynq）
- Worker处理任务
- gRPC调用Agent执行

**实时日志**:
- gRPC双向流接收Agent日志
- WebSocket推送到前端
- 日志持久化到数据库

**数据模型**:
```go
type Task struct {
    ID          uint   `gorm:"primarykey"`
    Name        string `gorm:"size:200;not null"`
    Type        string `gorm:"size:50"` // script/command/file
    Content     string `gorm:"type:text"` // 脚本内容或命令
    Targets     string `gorm:"type:text"` // JSON格式目标机器
    Status      string `gorm:"size:20;index"` // pending/running/success/failed
    Timeout     int    `gorm:"default:300"` // 超时时间（秒）
    CreatorID   uint   `gorm:"index"`
    CreatedAt   time.Time
    UpdatedAt   time.Time
}

type TaskLog struct {
    ID        uint   `gorm:"primarykey"`
    TaskID    uint   `gorm:"index;not null"`
    Hostname  string `gorm:"size:100;index"`
    Content   string `gorm:"type:text"` // 日志内容
    ExitCode  int    `gorm:"default:0"`
    CreatedAt time.Time
}
```

#### Agent端实现

**Agent功能**:
- gRPC长连接到Server
- 心跳保持连接
- 接收任务执行
- 实时上报日志

**执行安全**:
- 命令白名单检查
- 执行用户限制（非root）
- 超时控制
- 资源限制

**Agent部署**:
- 独立二进制文件
- systemd服务管理
- 配置文件（Server地址、认证信息）

#### 前端实现

**任务创建**:
- 目标机器选择（树形选择器）
- 脚本编辑（Monaco Editor）
- 参数配置

**执行监控**:
- 实时日志（WebSocket + 虚拟滚动）
- 进度条（成功/失败/进行中）
- 执行统计

**历史记录**:
- 任务列表
- 执行结果

## 九、总结

本模块设计方案详细描述了BigOps平台8个功能模块的设计思路、技术实现和API接口。各模块既相对独立，又通过统一的认证授权和数据模型实现了良好的集成。

- **安全性**: 全面的安全设计，保障系统安全
