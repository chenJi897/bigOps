# P1 任务表单类型约束优化方案

**状态**: draft  
**目标**: 解决任务创建/编辑中 `task_type` 与 `script_type` 语义重复、可选值冲突的问题。  
**问题来源**: 当前表单允许出现 `task_type=python` 但 `script_type=bash` 的非法组合，导致用户认知混乱与执行风险。

---

## 1. 现状问题

### 1.1 字段重复
- `task_type` 和 `script_type` 都在表达“怎么执行脚本”，语义高度重叠。
- 用户无法判断两者谁是主字段，容易填错。

### 1.2 非法组合未被拦截
- 当前 UI 可选值未联动，后端校验也不够严格。
- 典型错误：`task_type=python` + `script_type=bash`。

### 1.3 对后续能力扩展不友好
- 未来若增加 `powershell`、`ansible`、`sql` 等类型，双字段模型会指数增加组合复杂度。

---

## 2. 设计原则

- **单一事实源**：执行语言/引擎只保留一个主维度。
- **前后端双重约束**：前端联动约束 + 后端强校验，避免脏数据。
- **兼容渐进迁移**：不一次性破坏历史数据与旧接口。

---

## 3. 推荐方案（优先）

## 方案 A：以 `script_type` 为唯一执行类型，`task_type` 退化为“任务大类”

### 3.1 字段语义重定义
- `task_type`：任务大类（如 `script` / `file_transfer` / `api_call` / `workflow`）
- `script_type`：仅当 `task_type=script` 时必填（`bash` / `python` / `sh` / `powershell`）

### 3.2 表单联动规则
- 当 `task_type != script`：
  - 隐藏或禁用 `script_type`
  - `script_content` 可改为对应配置区域（例如文件分发参数）
- 当 `task_type = script`：
  - `script_type` 必填
  - `script_content` 必填
  - 可根据 `script_type` 切换编辑器高亮语言

### 3.3 后端校验规则（必须）
- `task_type=script` 且 `script_type` 不在允许集合 -> 400
- `task_type!=script` 但传入 `script_type` -> 400（或忽略并审计告警）
- 存储层拒绝非法组合，避免后续统计/调度污染

---

## 4. 备选方案（不优先）

## 方案 B：完全删除 `task_type`，统一为 `script_type`

- 优点：最简模型
- 缺点：不利于后续非脚本任务扩展（如文件分发/工作流）
- 结论：暂不采用

---

## 5. 迁移策略（建议两阶段）

### 阶段 1：兼容 + 强校验（本期）
- 保留现有字段
- 前端先做联动约束
- 后端新增组合校验（拒绝非法组合）
- 文档与 Swagger 明确组合关系

### 阶段 2：语义收敛（下期）
- 梳理历史数据并修复非法组合
- 补充 DB 约束（应用层 + 可选 CHECK 约束）
- API 响应中增加 `effective_script_type`（可选）帮助平滑过渡

---

## 6. 验收标准

- [ ] UI 不再出现 `task_type=python` / `script_type=bash` 这类冲突选择
- [ ] 后端接口对非法组合返回明确错误信息
- [ ] 历史任务列表可正常显示，不影响执行
- [ ] 新增/编辑任务的 E2E 用例覆盖合法与非法组合

---

## 7. 测试用例建议

1. `task_type=script, script_type=bash` -> 成功  
2. `task_type=script, script_type=python` -> 成功  
3. `task_type=script, script_type=空` -> 失败  
4. `task_type=file_transfer, script_type=bash` -> 失败  
5. 历史数据回显：旧任务打开编辑页应自动映射为合法状态或提示修复

---

## 8. 结论

优先执行 **方案 A**：保留 `task_type` 作为业务分类，`script_type` 作为脚本执行语言，并通过前后端联动约束消除冲突组合。  
该方案改动面小、可渐进落地，且能支撑后续任务中心扩展。

---

## 9. 前端“脚本内容补齐”专项（bash 优先）

> 目标：降低脚本编写门槛、减少空脚本/错脚本提交、提升执行成功率。  
> 范围：优先覆盖 `bash`，`python` 可复用同一机制扩展。

### 9.1 表单联动约束（第一优先）

- `task_type=script` 时：
  - 展示 `script_type` 下拉（仅允许 `bash` / `python` / `sh` 等白名单）
  - 展示 `script_content` 编辑区
- `task_type!=script` 时：
  - 隐藏或禁用 `script_type`
  - 隐藏 `script_content`，改为对应任务类型配置区
- 切换 `task_type` 或 `script_type` 时，若已有内容可能丢失，弹二次确认

### 9.2 bash 默认模板（第一批落地）

- 当用户选择 `script_type=bash` 且内容为空时，自动填充模板：

```bash
#!/usr/bin/env bash
set -euo pipefail

echo "[INFO] start at: $(date '+%F %T')"

# TODO: 在这里编写你的运维逻辑
# 示例：
# systemctl status nginx

echo "[INFO] done at: $(date '+%F %T')"
```

- 提供“一键恢复默认模板”按钮
- 新建任务默认给模板，编辑已有任务不强制覆盖

### 9.3 片段库（snippets）

- `bash` 常用片段：
  - 条件判断 `if/elif/else`
  - 循环 `for/while`
  - 函数定义与调用
  - 远程命令模板（预留）
  - 错误处理与退出码检查
- 交互方式：编辑器右上角下拉“插入片段”
- 插入行为：在光标处插入，不覆盖全文

### 9.4 基础静态校验（前端）

- 提交前做轻量校验并阻断：
  - `script_type=bash` 但内容为空
  - 缺少 shebang（可警告，不强阻断）
  - 明显危险命令（可配置黑名单）如 `rm -rf /`（强阻断或二次确认）
  - `task_type/script_type` 冲突组合（强阻断）
- 错误提示需可读：指出字段 + 修复建议

### 9.5 编辑器能力（可与现有实现平滑集成）

- 语法高亮：`bash` -> shell 高亮
- 行号、自动缩进、Tab 行为统一
- 可选：格式化按钮（仅做基础缩进，不做语义改写）
- 可选：快捷键说明（如 `Ctrl/Cmd+Enter` 运行预检查）

### 9.6 与后端契约对齐

- 前端的类型白名单与后端校验白名单必须一致（建议由后端下发配置）
- 前端所有校验只做“提前提醒”，最终以后端校验为准
- 接口错误统一回显到字段旁，避免“只弹 toast 看不懂”

### 9.7 验收标准（前端专项）

- [ ] 新建脚本任务选择 `bash` 时自动填充默认模板
- [ ] `task_type` 与 `script_type` 冲突时无法提交
- [ ] 空脚本不可提交
- [ ] 片段插入可在光标位置成功写入
- [ ] 编辑已有任务时，不会被默认模板覆盖

### 9.8 分期建议

- **P1-A（本期）**：联动约束 + bash 默认模板 + 基础校验  
- **P1-B（次期）**：片段库 + 编辑器增强 + 黑名单可配置化  
- **P2**：脚本 lint/静态分析、执行前模拟检查、模板市场

## 10. 开源 JS 方案选型（脚本编辑场景）

### 10.1 可直接采用的开源库

- **Monaco Editor**
  - 优点：VS Code 同款编辑体验、语言支持广、补全与片段能力强
  - 缺点：体积相对大，首屏加载需优化（懒加载/分包）
  - 适用：任务脚本编辑是高频核心交互时

- **CodeMirror 6**
  - 优点：轻量、模块化，按需引入更灵活
  - 缺点：高级能力需要更多自行组合
  - 适用：后台场景追求体积和可控性时

- **xterm.js**
  - 用途：日志展示区终端化（非编辑器）
  - 适用：任务执行日志、实时输出可视化

- **bash-parser / shell-quote（轻校验辅助）**
  - 用途：前端基础语法解析、命令拆分辅助校验
  - 注意：仅做“提前提示”，最终以后端校验为准

### 10.2 推荐组合（当前阶段）

- **P1-A 推荐**：`CodeMirror 6 + 自定义规则校验`
  - 原因：改造成本较低、包体更小、足够覆盖默认模板 + 片段 + 基础高亮
- **P1-B 可升级**：`Monaco Editor`
  - 当你们需要更强的补全、片段管理、编辑器体验一致性时再切换

### 10.3 接入建议

- 编辑器组件封装为 `ScriptEditor`，对外暴露：
  - `modelValue`
  - `language`（bash/python）
  - `insertSnippet(snippetKey)`
  - `validate()`
- 页面层只处理业务联动（`task_type/script_type`），编辑器层处理文本交互。

### 10.4 性能与工程化注意事项

- 编辑器必须懒加载（路由级或弹窗打开时加载）
- `bash` 与 `python` 语言包按需加载
- 片段和危险命令规则独立配置文件，便于后续运营调整

### 10.5 验收补充

- [ ] 首次打开任务创建页时，不因编辑器导致明显卡顿
- [ ] 切换 `bash/python` 时高亮语言正确切换
- [ ] 插入片段、模板恢复、基础校验都可稳定复现
