## 一、主流开源方案对比

| 项目 | Stars | 技术栈 | Agent 模式 | 通信方式 | 核心特性 | 适合规模 |
|------|-------|--------|-----------|----------|----------|----------|
| **Spug** | 11K | Python/Django + React | **无 Agent（SSH）** | SSH 协议 | 批量执行/在线终端/发布部署/监控/配置中心 | 中小 |
| **蓝鲸作业平台** | 1.2K | Java | **有 Agent（GSE）** | ZeroMQ + 私有协议 | 脚本执行/文件分发/定时任务/AI Agent | 大型 |
| **SaltStack** | 14K | Python | **有 Agent（Minion）** | ZeroMQ pub/sub | 远程执行/状态管理/编排/事件驱动 | 大型 |
| **Ansible** | 64K | Python | **无 Agent（SSH）** | SSH + WinRM | Playbook/模块化/幂等性/Inventory | 中大型 |
| **rce-agent** | 200+ | **Go** | **有 Agent** | **gRPC + TLS** | 命令白名单/双向 TLS 认证/轻量 | 嵌入式 |
| **go-grpc-agent-poc** | 小 | **Go** | **有 Agent** | **gRPC** | Master-Agent PoC/SSH 启动远端进程 | 演示 |

### 关键结论

1. **无 Agent（SSH）** — Spug/Ansible 模式，部署简单，但 SSH 连接管理复杂、实时日志能力弱、并发受限
2. **有 Agent（gRPC）** — 蓝鲸/rce-agent 模式，实时双向流、心跳检测、命令白名单、高并发
3. **BigOps 推荐方案**：**有 Agent + gRPC**，理由：
   - 已有 Go 技术栈，gRPC 生态成熟
   - 需要实时日志推送（gRPC 双向流天然支持）
   - 需要 Agent 心跳/状态监控
   - rce-agent 可直接参考其 TLS + 白名单设计

## 二、BigOps 任务执行中心架构设计

### 2.1 整体架构

```
┌─────────────┐   HTTP    ┌──────────────┐  gRPC Stream  ┌─────────────┐
│   前端 Vue   │ ◄──────► │  Core Server │ ◄────────────► │   Agent      │
│  (WebSocket) │          │  (Gin + gRPC)│               │ (目标主机)    │
└─────────────┘           └──────┬───────┘               └─────────────┘
                                 │                              ↑
                          ┌──────▼───────┐               ┌─────┴──────┐
                          │    MySQL     │               │   Agent    │
                          │   (任务记录)  │               │ (更多主机)  │
                          └──────────────┘               └────────────┘
```

### 2.2 通信协议选择

| 方案 | 优势 | 劣势 | BigOps 适用 |
|------|------|------|-------------|
| SSH | 无需 Agent，部署简单 | 连接管理复杂、实时日志弱 | ❌ |
| gRPC 双向流 | 实时日志、心跳、高性能、Go 原生 | 需要部署 Agent | ✅ **推荐** |
| WebSocket | 实时性好 | 浏览器场景，不适合 Agent 通信 | 仅前端推送 |

### 2.3 数据模型

```go
// Task 任务模型
type Task struct {
    ID          int64   // 主键
    Name        string  // 任务名称
    Type        string  // script/command/file
    Content     string  // 脚本内容或命令
    Timeout     int     // 超时秒数
    CreatorID   int64   // 创建人
    Status      string  // draft/ready
}

// TaskExecution 任务执行记录
type TaskExecution struct {
    ID          int64   // 主键
    TaskID      int64   // 关联任务
    Targets     string  // JSON: 目标主机 ID 列表
    Status      string  // pending/running/success/failed/partial
    StartedAt   *time.Time
    FinishedAt  *time.Time
    SuccessCount int
    FailedCount  int
    TotalCount   int
    CreatorID   int64
}

// TaskHostResult 单主机执行结果
type TaskHostResult struct {
    ID            int64
    ExecutionID   int64   // 关联执行记录
    AssetID       int64   // 目标主机
    Hostname      string  // 冗余
    IP            string  // 冗余
    Status        string  // pending/running/success/failed/timeout
    ExitCode      int
    Output        string  // stdout
    ErrorOutput   string  // stderr
    StartedAt     *time.Time
    FinishedAt    *time.Time
}

// Agent 注册信息
type AgentInfo struct {
    ID          int64
    AssetID     int64   // 关联 CMDB 资产
    Hostname    string
    IP          string
    Version     string
    Status      string  // online/offline
    LastHeartbeat *time.Time
}
```

### 2.4 gRPC Proto 定义（核心）

```protobuf
service TaskAgent {
    // Agent 注册 + 心跳（双向流）
    rpc Heartbeat(stream HeartbeatRequest) returns (stream HeartbeatResponse);

    // Server 推送任务给 Agent 执行
    rpc Execute(ExecuteRequest) returns (stream ExecuteResponse);

    // 文件传输
    rpc FileTransfer(stream FileChunk) returns (TransferResult);
}

message ExecuteRequest {
    string task_id = 1;
    string type = 2;      // script/command
    string content = 3;   // 脚本内容
    int32 timeout = 4;
}

message ExecuteResponse {
    string task_id = 1;
    string type = 2;      // stdout/stderr/exit/error
    bytes data = 3;
    int32 exit_code = 4;
}
```

### 2.5 Agent 设计要点

| 要点 | 方案 |
|------|------|
| 部署 | 单二进制文件 + systemd 服务 |
| 认证 | TLS 双向认证（参考 rce-agent） |
| 心跳 | gRPC 双向流，30s 间隔 |
| 命令控制 | 白名单 + 超时控制 + 非 root 执行 |
| 日志上报 | gRPC 流式返回 stdout/stderr |
| 自动注册 | 首次连接时自动关联 CMDB 资产（按 IP 匹配） |
| 升级 | Server 推送新版本 → Agent 自更新 |

### 2.6 前端实时日志

```
前端 ──WebSocket──► Core Server ──gRPC Stream──► Agent
                          ↓
                    聚合多主机日志流
                    推送给前端 WebSocket
```

## 三、与现有系统集成点

| 现有模块 | 集成方式 |
|----------|----------|
| **CMDB 资产** | 目标主机从 assets 表选择（复用 AssetRepository） |
| **服务树** | 按服务树节点批量选择执行目标 |
| **工单系统** | 工单审批通过后自动触发任务执行（ExecutionOrder） |
| **负责人** | 只有资产负责人或管理员可以对主机执行任务 |
| **审计日志** | 任务执行自动记录审计（who/when/what/where） |
| **Agent 状态** | 首页统计：Agent 在线/离线数 |

## 四、实施计划

1. gRPC Proto 定义 + Server/Client 生成
2. Agent 二进制（心跳 + 命令执行 + 日志流式返回）

## 五、技术选型总结

| 决策 | 选择 | 理由 |
|------|------|------|
| 通信协议 | gRPC 双向流 | Go 原生支持、实时日志、高性能 |
| Agent 认证 | TLS 双向证书 | 参考 rce-agent，安全性高 |
| 前端实时日志 | WebSocket | 浏览器原生支持，Core Server 聚合 gRPC 流后转发 |
| 任务队列 | 暂不引入 Asynq | V1 直接 goroutine 并发执行，后续按需加 |
| 目标选择 | 复用 CMDB assets | 不单独维护主机列表，Agent 关联 asset_id |
| Agent 部署 | systemd 单进程 | 中小规模足够，不需要 Docker |
