# BigOps 大运维平台 - 技术选型方案

**版本**: v1.0
**日期**: 2026-03-17
**状态**: 已定稿

## 一、项目概述

### 1.1 项目背景

BigOps 是一个面向中小团队（50人以下）的综合运维管理平台，旨在提供统一的运维能力，包括资产管理、任务执行、监控告警、工单管理等核心功能。

### 1.2 核心需求

- **用户规模**: 中小团队（50人以下）
- **部署架构**: 混合架构（核心模块单体 + 独立模块）
- **开发优先级**: 基础运维能力优先（模块01/02/03 → 模块04/05 → 模块06/07/08）
- **认证方式**: 本地账号密码
- **任务执行**: 仅支持Linux
- **监控数据源**: 自建Prometheus + 对接现有监控
- **K8s管理**: 暂不需要（模块06后期再加）
- **CICD平台**: 暂不需要（模块07后期再加）
- **数据库管理**: 仅MySQL管理（模块08）

### 1.3 技术方案选择

**选择方案A（轻量级快速启动方案）**

优先保证核心功能快速上线，技术栈简单成熟，降低学习和维护成本。

## 二、后端技术栈

### 2.1 核心框架

#### Web框架: Gin v1.9+
- **选型理由**: 轻量高性能，中间件生态丰富，学习曲线平缓
- **主要用途**: HTTP API服务、路由管理、中间件处理
- **性能指标**: 支持数万QPS，内存占用低

### 2.2 数据存储

#### 关系型数据库: MySQL 8.0+
- **用途**: 业务数据存储（用户、权限、CMDB、工单、任务等）
- **连接池配置**:
  - max_open_conns: 100
  - max_idle_conns: 10
  - conn_max_lifetime: 1小时

#### ORM框架: GORM v2
- **选型理由**: Go生态最成熟的ORM，支持自动迁移、关联查询、事务
- **主要功能**:
  - 数据库操作封装
  - 模型定义和关联
  - 自动迁移管理
  - 软删除支持

#### 缓存: Redis 7.0+
- **用途**:
  - 会话存储（JWT token）
  - 任务队列（Asynq）
  - 缓存热点数据
  - 分布式锁
- **客户端**: go-redis v9

### 2.3 认证授权

#### 认证: JWT (golang-jwt/jwt v5)
- **Token策略**:
  - access_token: 2小时过期
  - refresh_token: 7天过期
- **存储方式**: Redis（支持token撤销）
- **安全措施**:
  - 签名验证
  - 黑名单机制
  - 自动刷新

#### 权限管理: Casbin v2
- **模型**: RBAC（基于角色的访问控制）
- **存储**: MySQL（gorm-adapter）
- **控制粒度**:
  - API权限控制
  - 数据权限控制
  - 菜单权限控制

### 2.4 任务队列

#### Asynq v0.24+
- **选型理由**: 基于Redis的分布式任务队列，支持延迟任务、重试、优先级
- **主要功能**:
  - 异步任务执行
  - 定时任务调度
  - 任务重试机制
  - 任务优先级
- **队列配置**: 支持多队列（high/default/low）

### 2.5 服务通信

#### gRPC: google.golang.org/grpc v1.60+
- **用途**: 模块04任务执行中心的Server-Agent通信
- **协议**: Protobuf v3
- **特性**:
  - 双向流通信
  - 心跳检测
  - 自动重连
  - 高性能二进制传输

### 2.6 配置管理

#### Viper v1.18+
- **支持格式**: YAML/JSON/TOML
- **主要功能**:
  - 配置文件读取
  - 环境变量覆盖
  - 配置热加载
  - 多环境配置

### 2.7 日志系统

#### Zap v1.26+
- **选型理由**: 高性能结构化日志
- **配置策略**:
  - 开发环境: console输出，易读格式
  - 生产环境: json格式，便于采集
- **日志轮转**: lumberjack（按大小和时间轮转）
- **日志级别**: Debug/Info/Warn/Error/Fatal

### 2.8 API文档

#### Swag (swaggo/swag)
- **选型理由**: 从注释自动生成Swagger文档
- **集成方式**: gin-swagger中间件
- **访问路径**: /swagger/index.html
- **文档格式**: OpenAPI 3.0

## 三、前端技术栈

## 四、部署架构

### 4.1 混合部署方案

#### 核心模块（单体部署）
- **模块01**: 前后端底座
- **模块02**: 服务树和CMDB
- **模块03**: 自助工单
- **打包方式**: 单个Go二进制 + 前端静态文件
- **部署方式**: Docker Compose

#### 独立模块
- **模块04**: 任务执行中心（独立容器，需要高可用）
- **模块05**: 监控平台（独立容器）
- **模块08**: 数据库管理（独立容器）
- **部署方式**: Docker独立容器

### 4.2 容器化方案

#### Docker Compose服务列表
- **bigops-core**: 核心模块（01/02/03）
- **bigops-task**: 任务执行中心（04）
- **bigops-monitor**: 监控平台（05）
- **bigops-dbmgr**: 数据库管理（08）
- **mysql**: MySQL 8.0

- **prometheus**: Prometheus（可选，如果自建）

#### 网络配置
- **backend**: 内部网络（服务间通信）
- **frontend**: 外部网络（nginx对外）

## 五、安全设计

### 5.1 认证安全
- **密码加密**: Bcrypt（cost=10）
- **Token安全**: JWT签名 + Redis黑名单
- **会话管理**: 2小时过期 + 刷新机制
- **登录保护**: 失败次数限制（5次锁定10分钟）

### 5.2 权限安全
- **API权限**: Casbin中间件拦截
- **数据权限**: 查询时过滤（基于用户/角色）
- **操作审计**: 记录所有写操作

### 5.3 通信安全
- **HTTPS**: 生产环境强制HTTPS
- **gRPC**: TLS加密（可选）
- **敏感数据**: AES加密存储（数据库密码等）

### 5.4 执行安全
- **命令白名单**: 限制可执行命令
- **超时控制**: 防止任务无限执行
- **权限隔离**: Agent以普通用户执行

## 六、性能设计

### 6.3 并发控制
- **goroutine池**: 控制并发数
- **限流中间件**: API限流
- **任务队列**: 异步处理

## 八、技术风险与应对

### 8.1 任务执行安全性
- **风险**: Agent执行任意命令可能导致安全问题
- **应对**: 命令白名单、执行用户限制、审计日志

### 8.2 监控数据量大
- **风险**: Prometheus数据量大导致查询慢
- **应对**: 合理设置数据保留时间、使用Recording Rules预聚合

### 8.3 并发任务执行
- **风险**: 大量并发任务可能导致系统负载过高
- **应对**: 任务队列限流、goroutine池控制并发数

### 8.4 数据库性能
- **风险**: CMDB和日志表数据量大影响性能
- **应对**: 合理索引、分表策略、定期归档

## 十、总结

本技术选型方案采用轻量级、成熟稳定的技术栈，优先保证核心功能快速上线，适合中小团队快速启动。技术选型考虑了以下因素：

- **成熟度**: 选择社区活跃、文档完善的技术
- **学习成本**: 降低团队学习曲线
- **性能**: 满足中小团队性能需求
- **扩展性**: 为后续扩展预留空间
- **维护性**: 便于长期维护和升级

通过混合架构部署，既保证了核心模块的简单性，又为独立模块的扩展留有余地。后续可以根据实际使用情况，逐步优化和扩展功能，必要时可以平滑升级到微服务架构。
