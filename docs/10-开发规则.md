# BigOps 新模块接入统一规则

  ## 目标

  后续所有新模块必须一次性完成：

  - 功能代码
  - 菜单接入
  - 动态路由接入
  - 隐藏页 companion routes
  - Casbin API 权限
  - 数据级权限
  - admin 和普通角色验证

  禁止再采用“功能先跑起来，出问题再补”的方式。

  ---

  ## 1. 总原则

  1. 新模块接入必须一次性收口，不能只保证部分链路可用。
  2. 不能只验证 admin，必须验证普通角色。
  3. 不能依赖手工修当前数据库作为最终方案，必须给出可持续机制。
  4. 菜单、路由、权限、数据权限必须同步设计和交付。

  ---

  ## 2. 菜单规则

  1. 禁止在应用启动时 runtime seed 菜单。
  2. 默认菜单必须通过 migration 或 init SQL 提供，不允许通过服务启动时写库。
  3. migration 必须幂等，且能修正旧环境里的脏数据。
  4. migration 不仅要插入新菜单，还要修正以下字段：
     - `path`
     - `component`
     - `api_path`
     - `api_method`
     - `visible`
     - `status`
  5. 菜单管理页面只作为后续维护入口，不作为唯一初始化方式。

  ---

  ## 3. 目录菜单与页面菜单规则

  1. 目录节点（`type=1`）和页面节点（`type=2`）语义必须明确分离。
  2. 目录节点不能被当成真实页面路由处理。
  3. 前端动态路由生成时，目录节点即使 `path` 为空，也必须继续递归 children，不能直接跳过整棵子树。
  4. 侧边栏中的目录节点只负责展开/折叠，不应强依赖目录 `path` 跳转。
  5. 页面节点必须同时满足：
     - 有 `path`
     - 有 `component`
     - `component` 在前端 `viewModules` 中存在
  6. 如果页面菜单缺少合法组件映射，必须显式报错，不能静默失败。

  ---

  ## 4. 前端动态路由规则

  1. 每新增一个页面组件，必须同步更新 `router/index.ts` 中的 `viewModules`。
  2. 数据库中的菜单 `component` 必须与前端 `viewModules` key 完全一致。
  3. 如果模块存在创建页、详情页、执行页等隐藏入口，必须同步补 `companion routes`。
  4. `companion routes` 必须正确设置 `activeMenu`，高亮回落到主列表页。
  5. 所有页面组件必须显式定义：
     - `defineOptions({ name: 'XxxPage' })`
  6. 不允许出现菜单存在、页面文件存在，但因为缺少路由映射导致 404 的情况。

  ---

  ## 5. 菜单管理页面规则

  1. `Menus.vue` 中的 `componentOptions` 必须覆盖当前所有允许配置的页面组件。
  2. 新模块增加页面组件时，必须同步补充 `componentOptions`。
  3. 不允许出现“代码里支持该页面，但菜单管理页面无法选择”的情况。
  4. 菜单管理页面应服务于后续维护，而不是依赖开发手工记忆组件名。

  ---

  ## 6. 权限规则

  ### 6.1 两层权限必须分开

  1. 页面/API 访问权限：
     - 由 Casbin + 角色菜单控制
  2. 数据级权限：
     - 由 owner / 部门 / 服务树负责人 / 创建人等业务规则控制

  ### 6.2 具体要求

  1. 不允许把数据级权限错误地交给 Casbin 处理。
  2. Casbin 只负责“是否可以访问某类接口”，不负责“能看到哪些具体数据”。
  3. 如果某个资源支持负责人访问，则必须在 handler/service 层显式实现 owner/admin 判定。
  4. 如果普通角色可以进入页面但只应看到自己的数据，必须在列表接口层自动过滤，而不是依赖前端自己传参。

  ---

  ## 7. 资源模块统一要求

  适用于：

  - 主机资产
  - 云账号
  - 服务树
  - 工单
  - 任务中心
  - 后续所有带 owner 或负责人概念的模块

  ### 7.1 列表页

  1. 普通用户默认只看自己负责的数据。
  2. 管理员可以看全部数据。
  3. 不允许普通用户看到全部数据后再靠前端按钮隐藏操作。

  ### 7.2 详情与操作

  1. 详情页必须复用同一套 owner/admin 判定。
  2. 编辑、删除、变更历史等接口必须复用同一套 owner/admin 判定。
  3. 不允许出现列表能看、详情不能看，或详情能看、删除不能判定的割裂行为。

  ### 7.3 Casbin 与资源接口

  对于这类资源接口，必须明确选一种方案：

  #### 方案 A：菜单中配置完整 API 权限

  - 菜单配置 `api_path/api_method`
  - 角色分配菜单后同步 Casbin
  - 业务层再补充 owner 数据权限

  #### 方案 B：Casbin 白名单放行资源入口

  - 中间件对白名单资源接口先放行
  - 业务层强制做 owner/admin 数据权限校验

  禁止两种方案混乱拼接，导致：

  - 页面能打开
  - 接口却被 Casbin 误拦截
  - 或接口放开后又没有数据权限控制

  ---

  ## 8. 数据一致性规则

  1. 代码中的默认菜单定义、migration SQL、数据库现存数据三者必须保持一致。
  2. 任何 `path/component/api_path/api_method` 变更，都必须同步考虑旧环境迁移。
  3. 禁止“新代码改了，但旧数据库数据不修”。
  4. 禁止“当前环境手工改了能用，但其他环境仍然会复发”。

  ---

  ## 9. 验证要求

  每个新模块开发完成后，至少完成以下验证：

  ### 9.1 admin 验证

  1. 菜单显示正常
  2. 页面跳转不 404
  3. 主要接口可用
  4. 控制台无明显路由/组件警告

  ### 9.2 普通角色验证

  1. 只能看到被分配的菜单
  2. 页面打开不报无权限
  3. 如果有 owner 数据权限，只能看到自己的数据
  4. 非 owner 访问详情、编辑、删除时会被正确拒绝

  ### 9.3 工程验证

  1. `go build ./cmd/core` 通过
  2. `npm run build` 通过
  3. 浏览器控制台不能出现：
     - `Vue Router No match found`
     - `Element Plus prop validator` 警告
     - 其他明显运行时错误

  ---

  ## 10. 每次交付必须附带的内容

  以后每开发一个新模块，必须同时给出：

  1. 菜单 migration 文件
  2. 前端 `viewModules` 改动
  3. 是否需要 `companion routes`
  4. Casbin API 权限说明
  5. 数据级权限说明
  6. admin 验证结果
  7. 普通角色验证结果
  8. `go build ./cmd/core` 结果
  9. `npm run build` 结果

  ---

  ## 11. 禁止事项

  1. 禁止 runtime seed 菜单
  2. 禁止只修当前数据库数据而不修机制
  3. 禁止只测 admin 不测普通角色
  4. 禁止菜单、路由、权限不同步
  5. 禁止出现以下割裂状态：
     - 菜单能看到，点进去 404
     - 页面能打开，接口无权限
     - 有 owner 字段，但 owner 不能访问自己的数据
     - 路径更新了，但旧环境数据没迁移

  ---

  ## 12. 最终要求

  后续新增模块时，验收标准不再是“功能能不能跑”，而是：

  - 菜单是否正确
  - 路由是否完整
  - Casbin 是否正确
  - 数据权限是否正确
  - 普通角色是否可用
  - 旧环境是否可迁移
  - 是否具备可持续维护能力

  不允许继续通过临时补丁堆功能。
