## 一、调研总结

### 1.1 主流方案对比

| 项目 | 技术栈 | 工单模型 | 流程引擎 | CMDB联动 | 适合规模 |
|------|--------|----------|----------|----------|----------|
| **Ferry** | Go/Gin + Vue | 固定工单表 + 动态表单 | 自研（节点+连线 JSON） | 无 | 中小 |
| **蓝鲸ITSM** | Python/Django + Vue | ITIL标准（事件/变更/问题） | BPMN流程引擎 | 深度集成蓝鲸CMDB | 大型 |
| **NeatLogic** | Java | ITIL全流程 | 自研工作流 + 表单引擎 | 内置CMDB模块 | 中大型 |
| **easy-workflow** | Go（纯库） | 无业务模型，纯流程引擎 | 会签/驳回/并行网关 | 无 | 嵌入式 |
| **ServiceNow** | 商业 | CI关联工单（Incident/Change/Request） | BPMN | CMDB是核心，CI关联所有工单 | 企业级 |

### 1.2 关键设计决策

| 决策 | 选择 | 理由 |
|------|------|------|
| 工单类型 | **固定字段 + JSON 扩展字段** | 不引入表单设计器，V1 保持简单 |
| 流程引擎 | **简单状态机（代码驱动）** | V1 用固定流转规则，后续按需扩展 |
| 资源关联 | **通用 resource_type/resource_id** | 不限于资产，兼容后续 K8s/DB/Pipeline |
| 来源 | **source + source_event 字段** | 区分手动/监控/同步/CICD 等来源 |
| 审批流 | **V1 不做多级审批** | 先做"提交 → 处理 → 完成"三步流转 |
| 通知 | **V1 仅站内通知** | 不引入邮件/钉钉，先做最小可用 |
| 自动建单 | **统一入口 CreateTicketFromSource()** | 所有模块通过同一函数创建工单 |

## 二、工单数据模型

### 2.1 核心表

#### tickets — 工单主表

```go
type Ticket struct {
    ID              int64          `gorm:"primaryKey;autoIncrement" json:"id"`
    Title           string         `gorm:"size:200;not null" json:"title"`
    TicketNo        string         `gorm:"size:50;uniqueIndex;not null" json:"ticket_no"` // TK-20260324-0001
    TypeID          int64          `gorm:"index;not null" json:"type_id"`
    TypeName        string         `gorm:"-" json:"type_name,omitempty"`

    // 状态
    Status          string         `gorm:"size:20;not null;index" json:"status"`     // open/processing/resolved/closed/rejected
    Priority        string         `gorm:"size:20;not null" json:"priority"`         // low/medium/high/urgent

    // 内容
    Description     string         `gorm:"type:text" json:"description"`
    ExtraFields     string         `gorm:"type:json" json:"extra_fields"`            // 扩展字段 JSON

    // 来源（手动 or 自动建单）
    Source          string         `gorm:"size:20;not null;default:manual;index" json:"source"` // manual/monitor/sync/system/cicd
    SourceEventType string         `gorm:"size:50" json:"source_event_type"`         // alert/sync_failed/pipeline_failed/k8s_event
    SourceEventID   string         `gorm:"size:100;index" json:"source_event_id"`    // 来源事件唯一ID
    DedupeKey       string         `gorm:"size:200;index" json:"dedupe_key"`         // 去重键，防同一事件重复建单

    // 人员
    CreatorID       int64          `gorm:"index;not null" json:"creator_id"`
    CreatorName     string         `gorm:"-" json:"creator_name,omitempty"`
    AssigneeID      int64          `gorm:"index" json:"assignee_id"`
    AssigneeName    string         `gorm:"-" json:"assignee_name,omitempty"`

    // 部门
    SubmitDeptID    int64          `gorm:"index" json:"submit_dept_id"`
    SubmitDeptName  string         `gorm:"-" json:"submit_dept_name,omitempty"`
    HandleDeptID    int64          `gorm:"index" json:"handle_dept_id"`
    HandleDeptName  string         `gorm:"-" json:"handle_dept_name,omitempty"`

    // 通用资源关联（不限于资产，兼容 K8s/DB/Pipeline 等）
    ResourceType    string         `gorm:"size:50;index" json:"resource_type"`       // asset/cloud_account/service_tree/k8s_pod/database/pipeline
    ResourceID      int64          `gorm:"index" json:"resource_id"`
    ResourceName    string         `gorm:"size:200" json:"resource_name"`            // 冗余，展示用

    // 向后兼容：快捷访问（从 resource 自动填充）
    ServiceTreeID   int64          `gorm:"index" json:"service_tree_id"`

    // 处理结果
    Resolution      string         `gorm:"size:20" json:"resolution"`                // fixed/wontfix/duplicate/invalid/workaround
    ResolutionNote  string         `gorm:"type:text" json:"resolution_note"`         // 处理结果说明

    // SLA 预留
    SLADeadline     *LocalTime     `json:"sla_deadline"`                             // 处理时限
    RespondedAt     *LocalTime     `json:"responded_at"`                             // 首次响应时间

    // 时间
    ResolvedAt      *LocalTime     `json:"resolved_at"`
    ClosedAt        *LocalTime     `json:"closed_at"`
    CreatedAt       LocalTime      `json:"created_at"`
    UpdatedAt       LocalTime      `json:"updated_at"`
    DeletedAt       gorm.DeletedAt `gorm:"index" json:"-"`
}
```

#### ticket_types — 工单类型

```go
type TicketType struct {
    ID              int64          `gorm:"primaryKey;autoIncrement" json:"id"`
    Name            string         `gorm:"size:100;uniqueIndex;not null" json:"name"`
    Code            string         `gorm:"size:50;index" json:"code"`                  // incident/access/resource/change
    Icon            string         `gorm:"size:50" json:"icon"`
    Description     string         `gorm:"size:500" json:"description"`
    HandleDeptID    int64          `gorm:"default:0" json:"handle_dept_id"`
    DefaultAssignee int64          `gorm:"default:0" json:"default_assignee"`
    Priority        string         `gorm:"size:20;default:medium" json:"priority"`
    AutoAssignRule  string         `gorm:"size:50;default:manual" json:"auto_assign_rule"` // manual/resource_owner/service_owner
    Sort            int            `gorm:"default:0" json:"sort"`
    Status          int8           `gorm:"default:1" json:"status"`
    CreatedAt       LocalTime      `json:"created_at"`
    UpdatedAt       LocalTime      `json:"updated_at"`
    DeletedAt       gorm.DeletedAt `gorm:"index" json:"-"`

```

**AutoAssignRule 自动分派规则：**

| 规则 | 说明 | 实现 |
|------|------|------|
| `manual` | 手动指定处理人 | 默认行为 |
| `resource_owner` | 关联资源的负责人 | 读取 asset.owner_ids / cloud_account.owner_ids 第一个人 |
| `service_owner` | 服务树节点负责人 | 读取 service_tree.owner_id |
| `dept_default` | 处理部门默认人 | 读取 ticket_type.default_assignee |

#### ticket_activities — 工单活动流（替代 comment）

type TicketActivity struct {

### 2.2 编号规则

格式：`TK-{YYYYMMDD}-{4位序号}`，如 `TK-20260324-0001`

```go
func generateTicketNo() string {
    date := time.Now().Format("20060102")
    seq := getNextSeq(date) // Redis INCR 或 DB MAX+1
    return fmt.Sprintf("TK-%s-%04d", date, seq)
}
```

### 2.3 去重机制

自动建单时通过 `dedupe_key` 防止同一事件重复建单：

```go
// dedupe_key 示例：
// 监控告警: "monitor:alert:{alert_id}"
// 同步失败: "sync:failed:{cloud_account_id}:{date}"
// 流水线失败: "cicd:pipeline:{pipeline_id}:{run_id}"

func CreateTicketFromSource(req SourceTicketRequest) (*Ticket, error) {
    if req.DedupeKey != "" {
        existing, err := repo.GetByDedupeKey(req.DedupeKey)
        if err == nil && existing.Status != "closed" {
            return existing, nil // 已存在未关闭的工单，返回已有的
        }
    }
    // ... 创建新工单
}
```

## 三、工单状态机

### 3.1 状态流转图

```
                    ┌──────────────┐
                    │    open      │ 新建
                    └──────┬───────┘
                           │ assign（分配处理人）
                    ┌──────▼───────┐
            ┌───────│  processing  │ 处理中
            │       └──────┬───────┘
            │              │
     reject │    ┌─────────┼──────────┐
            │    │ resolve  │          │ comment/transfer
            │    │         │          │ (不改状态)
     ┌──────▼──┐ ┌────▼─────┐         │
     │rejected │ │ resolved │         │
     └─────────┘ └────┬─────┘         │
                      │ close          │
                ┌─────▼──────┐        │
                │   closed   │        │
                └─────┬──────┘        │
                      │ reopen        │
                      └───────────────┘
```

### 3.2 状态转换规则

```go
var ticketTransitions = map[string][]string{
    "open":       {"processing"},
    "processing": {"resolved", "rejected"},
    "resolved":   {"closed", "processing"},
    "rejected":   {"closed", "processing"},
    "closed":     {"processing"},
}
```

### 3.3 处理结果分类（关闭时必填）

| 结果 | 说明 |
|------|------|
| `fixed` | 已修复 |
| `workaround` | 临时方案 |
| `wontfix` | 不予处理 |
| `duplicate` | 重复工单 |
| `invalid` | 无效工单 |

### 3.4 权限控制

| 操作 | 谁能做 | 说明 |
|------|--------|------|
| 创建工单 | 所有登录用户 / 系统自动 | 选择类型、填写信息 |
| 分配处理人 | 管理员 / 处理部门成员 | open → processing |
| 处理（解决/驳回） | 当前处理人 | processing → resolved/rejected |
| 确认关闭 | 创建人 | resolved → closed（需选处理结果） |
| 重新打开 | 创建人 | closed/rejected → processing |
| 评论 | 创建人 + 处理人 | 不改变状态 |
| 转交 | 当前处理人 | 更换处理人，不改状态 |

## 四、通用资源关联

### 4.1 resource_type 枚举

工单不再只绑定 `asset_id`，而是通过 `resource_type + resource_id + resource_name` 关联任意资源：

| resource_type | 对应模块 | resource_id 含义 | resource_name 示例 |
|---------------|----------|------------------|-------------------|
| `asset` | CMDB | assets.id | `web-01 (10.0.1.1)` |
| `cloud_account` | CMDB | cloud_accounts.id | `阿里云生产环境` |
| `service_tree` | CMDB | service_trees.id | `业务线 / 产品A` |
| `k8s_cluster` | K8s（未来） | clusters.id | `prod-cluster` |
| `k8s_pod` | K8s（未来） | pods.id | `nginx-xxx-abc` |
| `database` | 数据库（未来） | databases.id | `prod-mysql-01` |
| `pipeline` | CICD（未来） | pipelines.id | `deploy-service-a` |
| （空） | 无关联 | 0 | - |

### 4.2 自动填充逻辑

```go
// 创建工单时，根据 resource_type 自动填充 service_tree_id
func autoFillFromResource(ticket *Ticket) {
    switch ticket.ResourceType {
    case "asset":
        asset, _ := assetRepo.GetByID(ticket.ResourceID)
        if asset != nil {
            ticket.ResourceName = fmt.Sprintf("%s (%s)", asset.Hostname, asset.IP)
            ticket.ServiceTreeID = asset.ServiceTreeID
        }
    case "cloud_account":
        account, _ := cloudAccountRepo.GetByID(ticket.ResourceID)
        if account != nil {
            ticket.ResourceName = account.Name
            ticket.ServiceTreeID = account.ServiceTreeID
        }
    // 后续模块扩展...
    }
}
```

### 4.3 默认分派规则

```go
// 自动分派处理人
func autoAssign(ticket *Ticket, ticketType *TicketType) {
    switch ticketType.AutoAssignRule {
    case "resource_owner":
        // 读取关联资源的负责人
        ownerID := getResourceOwner(ticket.ResourceType, ticket.ResourceID)
        if ownerID > 0 {
            ticket.AssigneeID = ownerID
        }
    case "service_owner":
        if ticket.ServiceTreeID > 0 {
            node, _ := serviceTreeRepo.GetByID(ticket.ServiceTreeID)
            if node != nil && node.OwnerID > 0 {
                ticket.AssigneeID = node.OwnerID
            }
        }
    case "dept_default":
        if ticketType.DefaultAssignee > 0 {
            ticket.AssigneeID = ticketType.DefaultAssignee
        }
    }
    // 如果分派成功，状态直接从 open 变为 processing
    if ticket.AssigneeID > 0 {
        ticket.Status = "processing"
    }
}

func getResourceOwner(resourceType string, resourceID int64) int64 {
    switch resourceType {
    case "asset":
        asset, _ := assetRepo.GetByID(resourceID)
        if asset != nil {
            ids := parseIntIDs(asset.OwnerIDs)
            if len(ids) > 0 { return ids[0] }
        }
    case "cloud_account":
        account, _ := cloudAccountRepo.GetByID(resourceID)
        if account != nil {
            ids := parseIntIDs(account.OwnerIDs)
            if len(ids) > 0 { return ids[0] }
        }
    }
    return 0
}
```

## 五、自动建单场景

### 5.1 统一入口

所有模块通过同一函数 `CreateTicketFromSource()` 自动建单：

```go
type SourceTicketRequest struct {
    Title           string
    TypeCode        string // "incident" / "change" / ...
    Description     string
    Priority        string
    Source          string // "monitor" / "sync" / "cicd" / "system"
    SourceEventType string // "alert" / "sync_failed" / "pipeline_failed"
    SourceEventID   string
    DedupeKey       string // 去重键
    ResourceType    string
    ResourceID      int64
    CreatorID       int64  // 0=系统用户
}
```

### 5.2 各模块场景

| 来源模块 | source | source_event_type | 触发时机 | dedupe_key 示例 |
|----------|--------|-------------------|----------|-----------------|
| **监控告警** | monitor | alert | Prometheus 告警触发 | `monitor:alert:{alert_fingerprint}` |
| **云同步** | sync | sync_failed | SyncRunner 返回 error | `sync:failed:{account_id}:{date}` |
| **K8s** | system | k8s_event | Pod CrashLoopBackOff | `k8s:crash:{namespace}/{pod}` |
| **数据库** | system | slow_query | 慢查询超阈值 | `db:slow:{db_id}:{date}` |
| **CICD** | cicd | pipeline_failed | 流水线失败 | `cicd:fail:{pipeline_id}:{run_id}` |
| **手动** | manual | （空） | 用户界面提交 | （空） |

### 5.3 去重逻辑

```
自动建单请求
  ↓
1. 有 dedupe_key？
  ├─ 否 → 直接创建
  └─ 是 → 查询同 dedupe_key 的未关闭工单
       ├─ 已存在 → 追加一条活动记录（"事件再次触发"），返回已有工单
       └─ 不存在 → 创建新工单
```

## 八、实施计划

### V1（最小可用版本）

1. 数据模型：Ticket + TicketType + TicketActivity（含全部扩展字段）
2. 工单类型 CRUD（5 个 API）
3. 工单 CRUD + 状态流转 + 转交（10 个 API）
4. 通用资源关联（resource_type/resource_id）
5. 自动分派规则（resource_owner / service_owner / dept_default）
6. 处理结果分类（关闭时必填 resolution）
7. 去重机制（dedupe_key）
8. 前端：工单列表 + 创建 + 详情 + 类型管理（4 个页面）
9. 首页统计扩展

### V2（流程增强 + 自动建单）

- 统一自动建单入口 `CreateTicketFromSource()`
- 监控告警自动建单（对接 Module 05）
- 同步失败自动建单（对接 SyncRunner）
- SLA 时效管理 + 超时告警
- 多级审批节点
- 邮件/钉钉通知

### V3（全面自动化）

- K8s 事件自动建单（对接 Module 06）
- CICD 流水线失败自动建单（对接 Module 07）
- 数据库慢查询自动建单（对接 Module 08）
- 工单触发任务执行（对接 Module 04）
- 工单统计报表 + SLA 达标率

## 十、架构决策备忘

| 决策 | 选择 | 理由 |
|------|------|------|
| resource_type 而非 asset_id | 通用关联 | 一次设计，后续 K8s/DB/Pipeline 零改动 |
| dedupe_key | 字符串索引 | 简单可靠，不需要分布式锁 |
| activity 而非 comment | 活动流 | 系统操作（自动建单/SLA预警）也需要记录 |
| auto_assign_rule 在 TicketType 上 | 类型级配置 | 不同类型不同分派策略，灵活度够 |
| resolution 在 Ticket 上 | 关闭时填 | 便于统计处理结果分布 |
| SLA 字段预留 | V1 建表不实现 | 字段先占位，V2 再加逻辑 |
| source + source_event | 双字段 | source 粗分类筛选，event 精确溯源 |
