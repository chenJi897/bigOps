## 一、调研总结

### 1.1 主流方案对比

| 项目 | 技术栈 | 工单模型 | 流程引擎 | CMDB联动 | 适合规模 |
|------|--------|----------|----------|----------|----------|
| **Ferry** | Go/Gin + Vue | 固定工单表 + 动态表单 | 自研（节点+连线 JSON） | 无 | 中小 |
| **蓝鲸ITSM** | Python/Django + Vue | ITIL标准（事件/变更/问题） | BPMN流程引擎 | 深度集成蓝鲸CMDB | 大型 |
| **NeatLogic** | Java | ITIL全流程 | 自研工作流 + 表单引擎 | 内置CMDB模块 | 中大型 |
| **easy-workflow** | Go（纯库） | 无业务模型，纯流程引擎 | 会签/驳回/并行网关 | 无 | 嵌入式 |
| **ServiceNow** | 商业 | CI关联工单（Incident/Change/Request） | BPMN | CMDB是核心，CI关联所有工单 | 企业级 |

### 1.2 关键设计决策

通过对比，BigOps 的工单系统采用以下策略：

| 决策 | 选择 | 理由 |
|------|------|------|
| 工单类型 | **固定字段 + JSON 扩展字段** | 不引入表单设计器，V1 保持简单。核心字段固定，特殊需求用 JSON |
| 流程引擎 | **简单状态机（代码驱动）** | 不引入 BPMN/流程设计器。V1 用固定流转规则，后续按需扩展 |
| CMDB 联动 | **工单关联资产 + 部门** | 工单绑定资产 ID，展示资产信息；提交部门 / 处理部门 |
| 审批流 | **V1 不做多级审批** | 先做"提交 → 处理 → 完成"三步流转，后续再加审批节点 |
| 通知 | **V1 仅站内通知** | 不引入邮件/钉钉，先做最小可用 |

## 二、工单数据模型

### 2.1 核心表

#### tickets — 工单主表

```go
type Ticket struct {
    ID              int64          `gorm:"primaryKey;autoIncrement" json:"id"`
    Title           string         `gorm:"size:200;not null" json:"title"`           // 工单标题
    TicketNo        string         `gorm:"size:50;uniqueIndex;not null" json:"ticket_no"` // 工单编号 TK-20260323-0001
    TypeID          int64          `gorm:"index;not null" json:"type_id"`            // 工单类型
    TypeName        string         `gorm:"-" json:"type_name,omitempty"`             // 关联查询

    // 状态
    Status          string         `gorm:"size:20;not null;index" json:"status"`     // open/processing/resolved/closed/rejected
    Priority        string         `gorm:"size:20;not null" json:"priority"`         // low/medium/high/urgent

    // 内容
    Description     string         `gorm:"type:text" json:"description"`             // 工单描述（Markdown）
    ExtraFields     string         `gorm:"type:json" json:"extra_fields"`            // 扩展字段 JSON

    // 人员
    CreatorID       int64          `gorm:"index;not null" json:"creator_id"`         // 创建人
    CreatorName     string         `gorm:"-" json:"creator_name,omitempty"`
    AssigneeID      int64          `gorm:"index" json:"assignee_id"`                 // 当前处理人
    AssigneeName    string         `gorm:"-" json:"assignee_name,omitempty"`

    // 部门
    SubmitDeptID    int64          `gorm:"index" json:"submit_dept_id"`              // 提交部门
    SubmitDeptName  string         `gorm:"-" json:"submit_dept_name,omitempty"`
    HandleDeptID    int64          `gorm:"index" json:"handle_dept_id"`              // 处理部门（如运维部）
    HandleDeptName  string         `gorm:"-" json:"handle_dept_name,omitempty"`

    // CMDB 关联
    AssetID         int64          `gorm:"index" json:"asset_id"`                    // 关联资产
    AssetHostname   string         `gorm:"-" json:"asset_hostname,omitempty"`
    ServiceTreeID   int64          `gorm:"index" json:"service_tree_id"`             // 关联服务树节点

    // 时间
    ResolvedAt      *LocalTime     `json:"resolved_at"`
    ClosedAt        *LocalTime     `json:"closed_at"`
    CreatedAt       LocalTime      `json:"created_at"`
    UpdatedAt       LocalTime      `json:"updated_at"`
    DeletedAt       gorm.DeletedAt `gorm:"index" json:"-"`
}
```

#### ticket_types — 工单类型

```go
type TicketType struct {
    ID              int64          `gorm:"primaryKey;autoIncrement" json:"id"`
    Name            string         `gorm:"size:100;uniqueIndex;not null" json:"name"`  // 如：故障报修、权限申请、资源申请
    Code            string         `gorm:"size:50;index" json:"code"`                  // incident/access/resource
    Icon            string         `gorm:"size:50" json:"icon"`                        // Element Plus 图标
    Description     string         `gorm:"size:500" json:"description"`
    HandleDeptID    int64          `gorm:"default:0" json:"handle_dept_id"`            // 默认处理部门
    DefaultAssignee int64          `gorm:"default:0" json:"default_assignee"`          // 默认处理人
    Priority        string         `gorm:"size:20;default:medium" json:"priority"`     // 默认优先级
    Sort            int            `gorm:"default:0" json:"sort"`
    Status          int8           `gorm:"default:1" json:"status"`                    // 1=启用 0=禁用
    CreatedAt       LocalTime      `json:"created_at"`
    UpdatedAt       LocalTime      `json:"updated_at"`
    DeletedAt       gorm.DeletedAt `gorm:"index" json:"-"`
}
```

#### ticket_comments — 工单评论/操作记录

```go
type TicketComment struct {
    ID           int64     `gorm:"primaryKey;autoIncrement" json:"id"`
    TicketID     int64     `gorm:"index;not null" json:"ticket_id"`
    UserID       int64     `gorm:"not null" json:"user_id"`
    UserName     string    `gorm:"-" json:"user_name,omitempty"`
    Content      string    `gorm:"type:text" json:"content"`                // 评论内容
    Action       string    `gorm:"size:20;not null" json:"action"`          // create/assign/comment/resolve/close/reject/reopen
    OldStatus    string    `gorm:"size:20" json:"old_status"`               // 状态变更前
    NewStatus    string    `gorm:"size:20" json:"new_status"`               // 状态变更后
    CreatedAt    LocalTime `json:"created_at"`
}
```

### 2.2 编号规则

格式：`TK-{YYYYMMDD}-{4位序号}`，如 `TK-20260323-0001`

```go
func generateTicketNo() string {
    date := time.Now().Format("20060102")
    // 从 Redis INCR 或 DB 查今天最大序号
    seq := getNextSeq(date)
    return fmt.Sprintf("TK-%s-%04d", date, seq)
}
```

## 三、工单状态机

### 3.1 状态流转图

```
                    ┌──────────────┐
                    │    open      │ 新建
                    └──────┬───────┘
                           │ assign（分配处理人）
                    ┌──────▼───────┐
            ┌───────│  processing  │ 处理中
            │       └──────┬───────┘
            │              │
     reject │    ┌─────────┼──────────┐
            │    │ resolve  │          │ comment
            │    │         │          │ (不改状态)
     ┌──────▼──┐ ┌────▼─────┐         │
     │rejected │ │ resolved │         │
     └─────────┘ └────┬─────┘         │
                      │ close          │
                ┌─────▼──────┐        │
                │   closed   │        │
                └─────┬──────┘        │
                      │ reopen        │
                      └───────────────┘
```

### 3.2 状态转换规则

```go
var ticketTransitions = map[string][]string{
    "open":       {"processing"},                    // 分配后自动变为处理中
    "processing": {"resolved", "rejected"},          // 处理人可解决或驳回
    "resolved":   {"closed", "processing"},          // 创建人确认关闭或重新打开
    "rejected":   {"closed", "processing"},          // 创建人接受驳回或重新打开
    "closed":     {"processing"},                    // 可重新打开
}
```

### 3.3 权限控制

| 操作 | 谁能做 | 说明 |
|------|--------|------|
| 创建工单 | 所有登录用户 | 选择类型、填写信息 |
| 分配处理人 | 管理员 / 处理部门成员 | open → processing |
| 处理（解决/驳回） | 当前处理人 | processing → resolved/rejected |
| 确认关闭 | 创建人 | resolved → closed |
| 重新打开 | 创建人 | closed/rejected → processing |
| 评论 | 创建人 + 处理人 | 不改变状态 |

## 四、CMDB 联动设计

### 4.1 工单关联资产

工单表的 `asset_id` 关联 `assets` 表。场景：

- **故障报修**：用户选择故障机器 → 工单自动填充 IP/hostname/服务树
- **权限申请**：用户选择目标服务器 → 处理人可查看资产详情
- **资源申请**：不关联资产，仅描述需求

前端实现：工单表单中嵌入资产搜索选择器（hostname/IP 模糊搜索）。

### 4.2 工单关联部门

- `submit_dept_id`：自动填充创建人所属部门
- `handle_dept_id`：根据工单类型自动填充默认处理部门，可手动修改

### 4.3 工单关联服务树

- `service_tree_id`：如果关联了资产，自动填充资产的 service_tree_id
- 支持按服务树节点筛选工单（查看某业务下所有工单）

### 4.4 统计联动

首页统计接口扩展：
- 待处理工单数
- 本周新建/已解决工单数
- 按工单类型分布
- 按部门分布

## 五、API 设计

### 5.1 工单类型管理（管理员）

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/ticket-types` | 工单类型列表 |
| GET | `/ticket-types/all` | 全量列表（下拉选择用） |
| POST | `/ticket-types` | 创建类型 |
| POST | `/ticket-types/:id` | 更新类型 |
| POST | `/ticket-types/:id/delete` | 删除类型 |

### 5.2 工单管理

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/tickets` | 工单列表（支持多条件筛选） |
| GET | `/tickets/:id` | 工单详情 |
| POST | `/tickets` | 创建工单 |
| POST | `/tickets/:id/assign` | 分配处理人 |
| POST | `/tickets/:id/process` | 处理（解决/驳回，含评论） |
| POST | `/tickets/:id/close` | 确认关闭 |
| POST | `/tickets/:id/reopen` | 重新打开 |
| POST | `/tickets/:id/comment` | 添加评论 |
| GET | `/tickets/:id/comments` | 评论/操作记录列表 |

### 5.3 筛选参数

```
GET /tickets?page=1&size=20
    &status=processing           # 状态筛选
    &priority=high               # 优先级
    &type_id=1                   # 工单类型
    &creator_id=5                # 创建人
    &assignee_id=3               # 处理人
    &handle_dept_id=2            # 处理部门
    &service_tree_id=10          # 服务树节点
    &keyword=磁盘                # 标题/描述模糊搜索
    &scope=my_created            # 视图维度: my_created/my_assigned/my_dept/all
```

## 七、实施计划

### V1（最小可用版本）— 预计 2-3 天

1. 数据模型：Ticket + TicketType + TicketComment
2. 工单类型 CRUD（5 个 API）
3. 工单 CRUD + 状态流转（9 个 API）
4. 前端：工单列表 + 创建 + 详情 + 类型管理（4 个页面）
5. CMDB 联动：工单关联资产选择器
6. 首页统计：待处理工单数

### V2（流程增强）— 后续迭代

- 多级审批节点
- 自定义表单字段（JSON Schema + 动态表单渲染）
- SLA 时效管理（响应时间/解决时间）
- 邮件/钉钉通知
- 工单模板

### V3（自动化）— 远期

- 工单触发任务执行（关联 Module 04）
- 工单自动创建（监控告警 → 自动生成故障工单）
- 工单统计报表

## 八、与现有系统的集成点

| 现有模块 | 集成方式 |
|----------|----------|
| 用户系统 | creator_id / assignee_id 关联 users 表 |
| 部门 | submit_dept_id / handle_dept_id 关联 departments 表 |
| CMDB 资产 | asset_id 关联 assets 表，展示 hostname/IP/状态 |
| 服务树 | service_tree_id 关联，支持按业务线筛选工单 |
| 审计日志 | 工单操作自动记录审计日志（已有 AuditLog 中间件） |
| 首页统计 | stats/summary 扩展工单统计字段 |
