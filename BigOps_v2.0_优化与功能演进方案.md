## 一、项目现状深度分析

### 1.1 已具备的坚实基础

**平台基础层**：
- 用户管理、角色管理、菜单管理、部门管理
- 成熟的 RBAC 权限体系（JWT + Casbin）
- 审计日志全覆盖
- 统一响应规范、日志体系（Zap）、配置管理

**运维核心模块**：
- **通知中台**：支持站内信、企业微信、钉钉、飞书、Webhook；具备模板、事件追溯、重试、配置管理等完整能力
- **监控告警**：数据源管理、告警规则、告警静默、OnCall 值班、告警事件处理流程
- **CMDB 与资产**：服务树、云账号同步、资产自动发现、资产变更历史
- **流程体系**：工单全生命周期管理 + 审批流 + 审批策略
- **CI/CD 基础**：项目管理、流水线、Webhook 触发

**Agent 执行引擎（核心亮点）**：
- 资源巡检（CPU 限制 + 内存巡检自杀机制）
- 脚本安全沙箱（scriptguard，支持 bash/python 危险命令检测）
- gRPC 双向流（Heartbeat + ReportOutput）
- 实时日志流式传输
- 进程组控制、超时强制终止、ulimit 资源限制
- 纯净指标采集（/proc + syscall，无第三方依赖）

### 1.2 当前存在的主要问题

**功能层面**：
- 各模块相对独立，缺乏统一的“任务中心”作为执行入口
- Agent 能力强大，但前端业务场景配套严重不足
- 告警事件处理体验不佳（你提供的多个设计图表明已有较好的设计思路，但实现滞后）
- 缺少知识库（Runbook）、统一变更管理、巡检系统等关键场景
- 缺乏统一的工作台和大屏视图

**体验与工程层面**：
- 前端页面主要集中在 Layout 和 NotificationConsole，其他核心模块前端实现不完整
- Agent 模块日志仍使用 `log.Printf`，与项目整体日志规范不一致
- 配置加载、代码结构、错误处理在 Agent 模块中存在不规范现象
- 可扩展性不足，缺少插件机制和任务编排引擎

**产品成熟度**：
- 当前更像“功能集合”，尚未形成完整的“智能运维闭环”

## 二、产品愿景与目标架构

### 2.1 产品愿景
将 BigOps 打造为**企业级一站式智能运维中台（AIOps Platform）**，实现“资产可知、状态可监、异常可知、问题可控、知识可复用、决策可辅助”。

### 2.2 核心能力目标
- **统一资产与服务树**：CMDB 作为全平台数据底座
- **智能监控告警**：从告警到自动处置的完整闭环
- **自动化执行引擎**：以统一任务中心为核心，充分发挥 Agent 能力
- **知识协同中心**：Runbook + 知识库 + 经验沉淀
- **可视化运营平台**：工作台、大屏、统计分析
- **智能化升级**：未来向根因分析、智能建议、预测性运维演进

## 三、功能演进三阶段路线图

### 阶段一：基础强化与体验升级（建议 6-8 周）

**目标**：解决当前最痛点的问题，让核心功能真正可用、好用、闭环。

**重点模块**：

1. **统一任务中心（最高优先级）**
   - 功能：任务模板、任务实例、实时日志、执行历史、批量操作、统计分析
   - 整合范围：Agent 脚本执行、巡检任务、CI/CD 执行、工单自动化
   - 交付物：任务中心前后端完整功能 + API

2. **智能告警中心（第二优先级）**
   - 基于你提供的设计图（alert-events-*.png 系列）进行重构
   - 实现表格视图 + 时间轴视图切换
   - 支持告警收敛、批量处理、关联知识、转工单、一键操作
   - 优化告警详情页和事件时间线

3. **巡检管理系统**
   - 巡检模板、巡检计划、执行记录、报告生成
   - 异常自动转工单
   - 巡检健康度看板
   - 直接复用现有 Agent 执行能力

4. **前端体验全面升级**
   - 重构 Layout.vue，增加清晰导航（工作台、任务中心、告警中心、巡检、知识库）
   - 统一使用 Pinia + Composition API
   - 优化通知中心、增加标签页管理

5. **技术规范统一治理**
   - Agent 模块日志全部切换为项目统一 Zap logger
   - 配置加载规范化
   - 代码结构优化（patrol、public_ip 等逻辑整理）
   - 错误处理和响应格式统一

### 阶段二：能力闭环与生态建设（2-4 个月）

**重点模块**：
- 知识库 & Runbook 系统（支持 Markdown、版本、搜索、关联告警）
- 变更管理系统（Change Management）
- 可视化仪表盘与个人工作台
- Agent 增强（文件分片传输、插件系统、自升级、离线缓存）
- 任务编排引擎（支持依赖、条件、并行）

### 阶段三：智能化与平台化（4-8 个月）

**重点方向**：
- AIOps 能力（智能根因分析、自动处置建议、异常预测）
- 插件市场与扩展生态
- 多租户支持
- 数据湖、报表中心、BI 分析
- OpenAPI 开放平台

## 四、阶段一详细功能规格（最重要部分）

### 4.1 统一任务中心规格

**核心页面**：
- 任务列表页（支持筛选、状态、类型、执行人、时间范围）
- 任务详情页（基本信息、执行记录、实时日志、输出结果）
- 任务模板管理页
- 执行历史统计分析页

**关键特性**：
- 支持多种触发方式（手动、定时、事件触发、Webhook）
- 实时日志流（WebSocket）
- 执行结果聚合展示（成功率、耗时分布）
- 失败自动重试策略配置

### 4.2 智能告警中心规格（重点）

**参考设计图**：alert-events-final.png、alert-events-v4.png、alert-events-table-v2.png、alert-events-top.png 等

**核心功能**：
- 视图切换：表格模式 / 时间轴模式
- 告警收敛分组展示
- 批量认领、批量解决、批量转工单
- 丰富筛选条件（服务树、负责人、级别、状态、时间等）
- 告警详情包含：事件时间线、关联资产、关联知识、处理记录
- 一键操作按钮（认领、解决、抑制、转工单）

### 4.3 巡检管理系统规格

- 巡检模板（脚本内容、执行节点、预期结果、阈值）
- 巡检计划（cron 表达式、手动触发、执行范围）
- 巡检报告（自动生成 PDF 或 Markdown 报告）
- 异常自动创建工单机制
- 巡检概览大盘（通过率、异常趋势、Top 问题）

## 五、技术优化清单（不改业务逻辑，只做规范统一）

1. **日志规范统一**
   - Agent 所有 `log.Printf` 改为项目统一的 logger + 结构化字段（agent_id、task_id 等）

2. **配置加载统一**
   - Agent 使用 `internal/pkg/config` 统一加载配置，移除直接 viper 调用

3. **代码结构优化**
   - patrol.go 改造为结构体 + 服务形式
   - public_ip 相关逻辑移动到 internal/agent 包内
   - executor.go 进一步抽象和增强

4. **公共组件与规范**
   - 统一错误码、响应格式
   - 统一分页、查询参数规范
   - 前端 API 调用统一错误处理

**第 7-8 周**：

## 七、风险与建议

**主要风险**：
- 前后端并行开发时接口联调效率
- 告警中心 UI 复杂度较高，需产品经理持续参与评审
- Agent 重构过程中可能引入新 Bug

**建议**：
- 采用“后端 API 先行 + 前端并行开发”的模式
- 每周举行方案评审与进度同步会议
- 优先确保“统一任务中心”作为阶段一核心交付物
